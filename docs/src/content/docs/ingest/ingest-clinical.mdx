If you have gotten this far, well done! Most of the hard work is done (hopefully).

Assuming that you have a configured CanDIG instance up and running and have access to a user with Site curator or Dataset curator credentials, you should be able to run through the instructions below.

## Ingesting OMOP formatted clinical data into CanDIGv3

Before you can use the ingest endpoint, you will need an authorized bearer token. Users with Site administration, Site curator, or dataset curator for the dataset/s being ingested are able to ingest those datasets. To get a bearer token:

1. Get a token by logging into the candig data portal as Site admin and copying the API token.

    a. Go to the icon in the top right of the screen and click the cog

    b. Click ‘ *** Get API Token’

     c. Click the token to copy the text

2. Open a terminal and save it into a variable called TOKEN

```bash
TOKEN=ey-pasted-jwt
```

:::danger
This token should be kept secure, it lasts for 30 mins
:::

3. Use the ingest endpoint to ingest data.

An example call to the ingest endpoint would look something like (note you will need to replace $CANDIG_URL with the root url of your instance):

```bash
curl -X POST $CANDIG_URL'/candig-api/v1/datasets/upload' \
  -H "Authorization: Bearer "$TOKEN \
  -H "Content-Type: multipart/form-data" \
  -F "file=@/absolute/path/to/omop_map.json;type=application/json"
```

If ingest worked correctly, you should get a response from the API giving you your queue_id to check the status of your ingest:

```bash
{"message": "File uploaded for processing.", "queue_id": "218961fad90b5c54", "url": "/v1/datasets/upload/status/218961fad90b5c54"}
```

Since ingest is an asynchronous process, it can take some time to complete depending on the size of your clinical data. To check on the status of your ingest, use the status endpoint:

```bash
curl -X 'GET' \
 $CANDIG_URL'/candig-api/v1/datasets/upload/status/218961fad90b5c54' \
 -H 'accept: application/json' \
 -H 'Content-Type: application/json' \
 -H 'Authorization: Bearer '$TOKEN
```

While the data is still being ingested, the status will read `"status": "still in queue"`. When ingest is complete, a successful ingest will have a summary of what was ingested, you should check the numbers are as expected. If any errors were found that would prevent ingest, they will be listed in the errors key in the response.

## Ingesting Sample Registration information into CanDIGv3

Since OMOP doesn't handle metadata related to samples, we will ingest this data into the system separately.

1. Using the MOHCCN formatted JSON that went into the mohccn-omop-etl tools, upload this to samples upload endpoint, along with the `site_id` that was used to create the OMOP-formatted json. 

e.g.

```bash
curl --request POST \
  --url $CANDIG_URL'/candig-api/v1/datasets/upload/samples?site_id=SITE_PM2C' \
  --header 'accept: application/json' \
  --header 'authorization: Bearer '$TOKEN \
  --header 'content-type: multipart/form-data' \
  --form file=@/absolute/path/to/input_mohccn_data.json
```

:::tip
If you aren't sure about your `site_id`, check the omop file for the prefix before `~` that is prepended to the dataset_id, alternatively, take a look at the `.env` file for the `MOHCCN_SITE_ID` in the `mohccn-omop-etl` repo where you ran the transformation
:::

2. You should receive confirmation of the upload and a queue id to monitor status like:

```bash
{"message": "File uploaded for processing.", "queue_id": "0080ef5d0a648a80", "url": "/v1/datasets/upload/status/0080ef5d0a648a80"}
```

2. You can then check the status of the upload the same as earlier with:

```bash
curl -X 'GET' \
 $CANDIG_URL'/candig-api/v1/datasets/upload/status/0080ef5d0a648a80' \
 -H 'accept: application/json' \
 -H 'Content-Type: application/json' \
 -H 'Authorization: Bearer '$TOKEN
```

And once complete, should receive a message like the following, with any errors that occurred in the `errors` key: 

```json
{
  "status": "Success",
  "file_name": "small_dataset_clinical_ingest.json",
  "file_size": 764056,
  "uploaded_at": "2026-03-16 22:09:53 UTC",
  "site_id": "SITE_PM2C",
  "data_type": "samples",
  "processed_at": "2026-03-16 22:09:54 UTC",
  "ingested_count": 252,
  "ingested_items": [
    "SITE_PM2C~SYNTH_01~SAMPLE_0001",
    "SITE_PM2C~SYNTH_01~SAMPLE_0002",
    "SITE_PM2C~SYNTH_01~SAMPLE_0003",
    "SITE_PM2C~SYNTH_01~SAMPLE_0004",
    ...
  ],
  "errors": null
}
```
